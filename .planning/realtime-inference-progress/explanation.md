# 实时推理进度解释

## 为什么要做底部进度展示

在线 nnUNetv2 推理不是瞬时请求，而是一个长耗时任务。用户最关心的不是“按钮点下去了吗”，而是：

- 现在做到哪一步了
- 是否还在运行
- 什么时候会结束
- 如果卡住，是正常慢还是出错了

所以需要一个结构化的底部进度区域，而不是只显示几行普通日志。

## 进度信号从哪里来

当前系统已经有后端 SSE：

```text
/api/segment/jobs/{job_id}/events
```

前端通过 `EventSource` 订阅后，会收到阶段事件，例如：

- `progress`
- `complete`
- `error`
- `heartbeat`

这意味着前端不需要自己猜推理状态，而是直接使用后端阶段事件。

## 为什么不能伪造真实百分比

nnUNetv2 主阶段通常是最耗时的部分，但其内部 patch 级进度并没有可靠暴露给前端。如果前端自己平滑推进百分比，视觉上会更流畅，但会误导用户，以为那是模型真实进度。

因此当前采用：

- **后端阶段百分比**：作为真实进度来源
- **心跳事件**：证明任务仍在运行
- **已耗时**：让用户知道没有卡死
- **阶段日志**：显示任务当前所在环节

## 前端怎么显示

前端维护结构化 `inferenceTimeline`，每收到 SSE 事件就写入一条记录。

底部会显示：

- 当前阶段
- 进度条
- Job ID
- 推理模式
- 已耗时
- 最近几条阶段日志

如果缓存命中，会快速显示完成态；如果失败或取消，会保留错误摘要和日志尾部。

## 后端心跳的作用

长耗时阶段可能在几分钟内百分比不变。为了避免用户误以为系统卡死，后端会定期发心跳事件，携带：

- 已耗时
- 当前阶段名
- 资源快照
- heartbeat 标记

前端收到心跳后会刷新已耗时和活跃状态，但不伪造更高的完成度。

## 验证口径

验收时应该观察：

1. 点击运行后底部立即出现进度条。
2. 运行中能看到当前阶段和已耗时。
3. 长时间阶段不会静默无反馈。
4. 完成后到 100%。
5. 失败和取消时能显示错误或终止状态。
6. 缓存命中时能快速回填完成态。

## 与标签验证的关系

进度展示和 validation 是两条不同链路：

- 进度展示负责告诉你推理还在不在跑。
- validation 负责告诉你结果对不对。

前者通过 SSE，后者通过标签和 remap 逻辑。

## 与量化功能的关系

量化功能不会影响进度条本身。它只是在推理完成并回填 result mask 后，在前端额外做一次 mask 统计。

因此完整顺序是：

```text
SSE 进度 -> 推理完成 -> 下载结果 -> 回填 mask -> 量化计算 -> 报告导出
```

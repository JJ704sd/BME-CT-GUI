# 局域网 API 连接与内网穿透规划

**项目：** `segmentation-gui-prototype`  
**目标：** 让前端 GUI 能在局域网访问后端 API，并为后续远程访问 / 内网穿透预留可行方案。  
**当前日期：** 2026-05-30

## 1. 背景

当前项目已经具备本地前后端联调能力：

- 前端开发服务默认只监听 `127.0.0.1:5173`
- 后端 FastAPI 默认也按本机调试方式启动
- 前端 `API_ENDPOINT` 目前是硬编码本机地址
- 后端 CORS 目前也只放行本地回环相关来源
- 推理链路包含上传、SSE 进度、结果下载、取消任务，属于长连接 + 大文件传输场景

这意味着：

1. **局域网内其他设备** 还不能稳定直接访问这套 GUI / API。
2. **远程外网访问** 更不能直接暴露后端，需要一个更安全、更稳定的接入层。

## 2. 目标拆分

### 2.1 局域网目标

在同一局域网内，另一台电脑或平板可以：

- 打开前端页面
- 访问后端 `/api/health`、`/api/models`、`/api/segment/jobs`
- 正常上传 CT、接收 SSE、下载结果
- 不需要手工改代码就能切换后端地址

### 2.2 内网穿透目标

在不暴露后端到公网裸端口的前提下，让外部网络也能访问这套服务，并尽量满足：

- 访问地址稳定
- 传输通道安全
- 不影响 SSE 长连接
- 不破坏大文件上传 / 下载
- 便于运维和回滚

## 3. 现状判断

### 3.1 代码层面的限制

当前主要限制有三类：

1. **前端地址硬编码**
   - `src/main.tsx` 里 `API_ENDPOINT` 还是本机地址

2. **后端绑定地址偏保守**
   - 默认适合本机开发，不适合局域网其他机器直连

3. **跨域策略过窄**
   - 目前只适合 `localhost / 127.0.0.1` 场景

### 3.2 业务层面的限制

这套应用不是普通 JSON 小 API，而是：

- 需要上传 CT 文件
- 需要 SSE 持续推送任务状态
- 需要下载推理结果
- 可能持续运行很久

所以方案不能只看“能不能 ping 通”，要看：

- 上传是否稳定
- SSE 是否会断
- 下载是否会超时
- 浏览器跨域是否干净
- 是否方便部署和维护

## 4. 推荐路线

## 路线 A：先做局域网直连，再考虑内网穿透

这是最稳妥的路径。

### 为什么先做局域网

- 改动小
- 可快速验证前后端分离后在非本机环境下是否能用
- 能提前暴露 CORS、地址配置、SSE、上传下载等问题
- 后续内网穿透其实是在“同一套地址配置”上换入口，不必重做逻辑

### 结论

**推荐先把 API 入口做成可配置，再把后端改成可绑定局域网地址。**

---

## 5. 局域网方案设计

### 5.1 前端：把 API 地址改成可配置

#### 方案

不要再把后端地址写死在 `src/main.tsx` 中，而是改成：

- 开发环境可从环境变量读取
- 生产环境可从部署配置读取
- 最终形成一个稳定的 `API_ENDPOINT`

#### 推荐做法

优先级从高到低：

1. `VITE_API_ENDPOINT`
2. 页面注入的运行时配置
3. 默认回退到本机地址

#### 结果

同一份前端构建包，可以通过不同配置连接：

- 本机后端
- 局域网后端
- 内网穿透后的远程后端

---

### 5.2 后端：监听局域网地址

#### 方案

把后端从只监听 `127.0.0.1` 改成可选监听：

- `127.0.0.1`：本机调试
- `0.0.0.0`：局域网可访问

#### 建议

用环境变量控制：

```bash
SEGMENTATION_BIND_HOST=0.0.0.0
SEGMENTATION_BIND_PORT=8000
```

然后在启动命令中显式指定。

---

### 5.3 后端：放行局域网来源的 CORS

#### 方案

当前 CORS 只适合 localhost。局域网访问时，浏览器来源会变成：

- `http://192.168.x.x:5173`
- `http://10.x.x.x:5173`
- 可能还有自定义域名

因此需要把 CORS 改成可配置白名单，而不是只允许回环。

#### 推荐策略

- 默认仍保守
- 通过环境变量显式放行可信来源
- 不建议长期使用全开放 `*` 作为正式部署默认值

例如：

```bash
SEGMENTATION_ALLOWED_ORIGINS=http://192.168.1.20:5173,http://192.168.1.20:4173
```

如需支持局域网段，也可以用 origin regex 做有限放行，但仍建议配置化。

---

### 5.4 前后端分离的部署方式

#### 开发期

- 前端仍用 Vite 开发服务器
- 后端监听 `0.0.0.0:8000`
- 前端通过局域网 IP 指向后端

#### 部署期

更推荐：

- 前端构建成静态文件
- 由 Nginx 或其他静态服务托管
- 后端单独跑 FastAPI
- 前端通过配置指向后端 API

这样更适合局域网分享和后续穿透。

## 6. 内网穿透方案对比

这里分成三种常见路线。

### 6.1 方案 1：Tailscale / WireGuard 类 VPN

#### 适用场景

- 只需要自己和团队成员访问
- 访问设备数量少
- 希望尽量安全、少暴露公网面

#### 优点

- 安全性高
- 不需要自己维护公网端口暴露
- 适合医疗影像这种敏感数据场景
- 内网访问体验接近局域网

#### 缺点

- 客户端通常要安装 VPN / Mesh 工具
- 对“纯浏览器访问”的友好度稍弱
- 若要给不方便安装客户端的人访问，体验一般

#### 结论

**如果是内部团队使用，我更推荐这一类方案。**

---

### 6.2 方案 2：frp / 自建 VPS 反向代理

#### 适用场景

- 需要一条固定公网入口
- 希望浏览器直接打开，无需额外 VPN 客户端
- 可以接受自己维护一台中转服务器

#### 优点

- 对浏览器最友好
- 可以把前端和后端都通过一个公网域名访问
- 方便做 HTTPS 和统一入口

#### 缺点

- 需要一台公网 VPS
- 需要自己维护 frp 配置、守护进程、证书、反代规则
- 安全责任更重

#### 结论

**如果需要“任何能上网的地方直接打开网页”且能接受运维成本，这个方案很实用。**

---

### 6.3 方案 3：Cloudflare Tunnel / 类托管穿透

#### 适用场景

- 希望少运维
- 希望快速做一个公网入口
- 能接受依赖第三方平台

#### 优点

- 配置相对简单
- 免公网端口暴露
- 常见于临时演示或低运维场景

#### 缺点

- 受第三方平台约束
- 需要确认数据合规和访问策略
- 医疗影像场景通常要谨慎评估

#### 结论

适合作为演示入口，不建议直接作为敏感医疗数据的唯一长期方案。

## 7. 推荐决策

### 如果优先考虑安全和内部使用

**推荐顺序：**

1. 局域网直连
2. Tailscale / WireGuard
3. frp / Cloudflare Tunnel

### 如果优先考虑“任意浏览器直接打开”

**推荐顺序：**

1. 局域网直连
2. frp + HTTPS 反代
3. Cloudflare Tunnel

### 我的建议

对于当前项目，最可行的组合是：

- **第一阶段：局域网直连**
- **第二阶段：Tailscale / WireGuard 作为首选内网穿透**
- **第三阶段：如果需要纯公网浏览器入口，再评估 frp**

原因：

- 医疗影像数据敏感，安全优先
- 现有系统已经有长任务、上传、下载和 SSE，不适合粗暴暴露端口
- VPN / Mesh 方案最容易把“远程访问”做成“像局域网一样访问”

## 8. 建议的实施步骤

### Phase 1：把地址配置化

需要做的事：

- 前端 `API_ENDPOINT` 改成环境变量/运行时配置
- 后端允许读取 `SEGMENTATION_BIND_HOST` / `SEGMENTATION_BIND_PORT`
- 后端 CORS 改成可配置白名单
- README 增加局域网连接示例

验收：

- 另一台局域网电脑可以打开前端
- 前端能正常调用 `/api/health`
- 能成功创建推理 job
- SSE 不报跨域错误
- 结果下载可用

---

### Phase 2：增加“部署模式”配置

可以把运行方式分成：

- `local-dev`
- `lan-dev`
- `lan-prod`
- `tunnel-prod`

每种模式只切换配置，不改业务逻辑。

建议抽出一个轻量配置层，集中管理：

- API 地址
- CORS 白名单
- 绑定地址
- 证书 / 反代地址

---

### Phase 3：接入内网穿透

如果选择 VPN / Mesh：

- 保持后端仍监听局域网地址
- 通过 Tailscale / WireGuard 分配的内部地址访问
- 前端 API 地址指向该内部地址

如果选择 frp：

- 在有公网 IP 的 VPS 上部署 frps
- 本地机器部署 frpc
- 反向映射前端 / 后端端口
- 在外部通过域名访问静态前端和 API

---

### Phase 4：加固

无论哪种穿透方案，都建议补充：

- HTTPS
- 访问控制
- 日志审计
- 资源限制
- 上传超时策略
- SSE 保活检查

## 9. 风险点

### 9.1 CORS 问题

局域网下最常见错误是：前端能打开，API 不能调用。

原因通常是：

- 前端 origin 不在白名单
- 后端仍只监听 localhost
- 前端 API 地址未切换

### 9.2 SSE 断连

如果穿透层不稳定，推理进度流容易断。

建议：

- 让客户端能自动重新连接
- 后端提供 job state 查询兜底
- SSE 只负责实时体验，不作为唯一状态源

### 9.3 大文件传输超时

CT 体数据上传可能很大，穿透层或反代需要确认：

- 请求体大小限制
- 上传超时
- 连接保持时间

### 9.4 安全风险

医疗影像数据不适合裸奔公网。

最低要求：

- 不要直接暴露未授权公网端口
- 外部访问最好有认证或隧道访问控制
- 日志中不要写入敏感文件内容

## 10. 建议的最终方案

### 最推荐

**局域网直连 + Tailscale / WireGuard**

这是当前最平衡的方案：

- 实现成本低
- 安全性高
- 对浏览器影响小
- 适合医疗影像类敏感数据

### 次推荐

**局域网直连 + frp + HTTPS**

适合必须对外提供浏览器入口的场景，但运维成本更高。

3. 本地局域网验证一轮
4. 再决定用 Tailscale / WireGuard 还是 frp
5. 最后补文档和部署命令

# 高分辨率 CT 推理优化解释

## 为什么需要这个说明

2026-05-31 的 AMOS CT 在线推理（768×768×103）已经完成，job `ad3d14eba3de` 使用 `fast` profile 得到 `mean_dice=0.77724`。该轮说明高分辨率输入会显著拉长本地 2D nnUNet 推理时间，并且 fast profile 只适合作为预览/演示证据，不能替代 AMOS `quality` 正式质量基线。本文档解释高分辨率输入导致推理速度下降的原因，以及后续可评估的优化路径。

## 核心问题

### 1. 输入分辨率与计算量的关系

当前 AMOS CT 的分辨率为 768×768×103，而标准 AMOS 数据集通常为 512×512×N。

```text
标准输入面积: 512 × 512 = 262,144 像素
当前输入面积: 768 × 768 = 589,824 像素
面积比: 589,824 / 262,144 ≈ 2.25 倍
```

对于 2D nnUNet 模型，每层切片需要独立推理。面积增加 2.25 倍意味着每层的计算量也增加约 2.25 倍。

### 2. 2D 模型处理 3D 数据

当前使用的模型是 `nnUNetTrainer__nnUNetPlans__2d`，它将 3D CT 数据按切片逐层处理：

```text
输入: 768×768×103
处理方式: 103 层 × 每层 768×768 推理
```

相比 3D 模型一次处理整个 volume，2D 模型在处理高分辨率切片时效率较低。

### 3. GPU 资源限制

RTX 4060 Laptop GPU 的限制：

```text
显存: 8GB
功率: 40W（受限于笔记本散热，实际约 27W）
计算单元: 3072 CUDA cores
```

推理过程中：

- 显存占用 95%（约 7.6GB / 8GB）
- GPU 利用率 100%
- 功率受限于散热，无法达到满功率

### 4. 模型复杂度

当前模型架构：

```text
ResidualEncoderUNet
8 个编码/解码阶段
约 10.2M 参数
patch size: 640×640
```

高分辨率输入需要更多的滑动窗口（sliding window）步骤来覆盖整个切片。

### 5. 重采样开销

nnUNetv2 在推理前需要将输入重采样到模型的 patch size：

```text
输入: 768×768
模型 patch: 640×640
需要重采样 + 滑动窗口覆盖
```

## 速度瓶颈总结

| 瓶颈因素 | 影响程度 | 说明 |
|---|---|---|
| 输入分辨率 | 高 | 面积增加 2.25 倍，直接增加计算量 |
| GPU 显存 | 中 | 接近上限，可能限制 batch size |
| GPU 功率 | 中 | 笔记本散热限制，无法满功率运行 |
| 模型复杂度 | 中 | 8 阶段网络，每阶段需计算 |
| 2D 模型 | 中 | 逐切片处理，无法利用 3D 上下文 |
| 重采样 | 低 | 坐标转换和插值计算 |

## 优化方向

### 短期：预降采样

在推理前对高分辨率 CT 进行降采样：

```python
# 示例：将 768×768 降采样到 512×512
import numpy as np
from scipy.ndimage import zoom

# 假设 ct_data shape 为 (768, 768, 103)
zoom_factors = (512/768, 512/768, 1.0)
ct_downsampled = zoom(ct_data, zoom_factors, order=1)
```

优点：
- 可能显著缩短推理时间，具体收益需用同一病例对照验证
- 实现简单，可在前端或后端完成

缺点：
- 可能丢失细节信息
- 需要在结果回填时上采样到原始分辨率

### 中期：使用 3D 模型

3D 模型可以利用体数据的上下文信息，可能在高分辨率输入上更高效：

```text
3D 模型: 一次处理整个 volume
2D 模型: 逐切片处理
```

但 3D 模型需要更多显存，可能需要更强大的 GPU。

### 长期：模型架构优化

- 训练支持更高分辨率的模型
- 使用注意力机制减少计算量
- 模型剪枝和量化

## 与现有系统的集成

当前系统已经支持：

- `quality` / `fast` 推理模式切换
- `runtime_target=local|server` 运行位置选择
- 推理结果缓存

高分辨率优化可以作为新的推理选项集成：

```text
inference_resolution=original|downsampled_512|downsampled_256
```

## 验证标准

优化后需要验证：

1. 推理时间是否显著缩短
2. 分割质量（Dice）是否可接受
3. 结果是否需要上采样到原始分辨率
4. 前端显示是否正常
